# ebay-auto-seller — 設計書（ロードマップ・実装方針）

最終更新: 2026-03-28

## 1. 目的とスコープ

本リポジトリは、eBay 向けの自動販売支援（在庫・価格・注文・調査）を、**バックエンド API**、**バッチ／イベント駆動ワーカー**、**Web UI**、**インフラ（AWS）** で提供することを目的とする。

この文書は、**実装の優先順位**と**各波のゴール**を固定し、個別 PR の議論と齟齬を減らすための設計書である。

---

## 2. 実装方針（原則）

| 原則 | 内容 |
|------|------|
| **ドメイン優先** | まずワーカーでデータの取得・更新が一貫することを優先し、API／Web はその上に載せる。 |
| **ワーカー深掘りの順序** | 第 1 波のサブ項目は **固定順** で実装する（後述）。 |
| **インフラは一気通貫** | 第 3 波では CDK のスタック定義にとどまらず、**デプロイ先 AWS 環境の構築**と**繰り返しデプロイ可能な手順（CI／`cdk deploy` 等）**までを同一マイルストーンに含める。 |
| **環境分離** | 少なくとも **検証用（dev）と本番（または staging）** を区別し、シークレット・DB・スケジュールは環境ごとに切り替え可能に設計する。 |
| **セキュリティ** | テナント境界、外部 HTTP（SSRF 対策）、トークン保存は既存方針（暗号化・許可ホスト等）を継承する。 |

---

## 3. 全体ロードマップ（波）

### 第 1 波 — ワーカー層の深掘り

**ゴール:** イベント入力だけに依存せず、注文・供給元スナップショットが運用で回る状態に近づける。

**実装順（この順で着手する）:**

1. **`orders-sync`: eBay REST による注文取り込み**  
   - Sell Fulfillment 等の REST API から注文を取得し、既存 `orders` への **冪等 upsert**（例: `tenant_id` + `ebay_order_id`）を行う。  
   - トリガ: EventBridge スケジュールまたは手動 invoke。  
   - 認証: 既存 eBay OAuth／トークンストアを利用。必要スコープを Fulfillment 向けに整理。  
   - `job_runs` で成否・エラーを記録。  
   - テスト: HTTP はモックし、マッピング・分岐をユニットテスト化。

2. **供給元アダプタの具体化**  
   - `sourcing_scan` のレジストリ上、プレースホルダ（例: `amazon` / `walmart` / `target`）を、認証・レート制限・レスポンスマッピング付きの実装へ置き換える。  
   - `source_type` → フェッチャのレジストリパターンは維持する。  
   - **実装方針（現状）:** 各 EC の公式 API をワーカー内で完結させず、テナント（または連携先）の **HTTPS プロキシ** が `http_json` と同一の JSON 契約を返す前提。行の `source_url` または env の URL テンプレート＋`external_product_id` で GET 先を決定する。

3. **`http_json` の運用強化（任意・必要に応じて）**  
   - テナントごとの契約（署名 URL、追加ヘッダ、固定出口との組み合わせ等）を README・設定に反映。

---

### 第 2 波 — API（および必要最小限の Web）

**ゴール:** ワーカーが書き込んだデータを、クライアントから参照・最小限の操作ができる。

- **Read 中心:** 調査候補、出品、注文、監査ログ等の読み取り API。  
- **Write 最小:** ワンクリック出品フロー等、ドメインに必要な最小の書き込み。  
- **Web:** API に追随する薄い画面（候補調査、出品管理、注文／利益ダッシュボード）は第 2 波後半または並行で拡張。

詳細エンドポイントは `services/api/README.md` の「Expected responsibilities」と整合させる。

---

### 第 3 波 — インフラ ＋ デプロイ先の構築

**ゴール:** 「ローカルと CI だけ」ではなく、**実運用に載せる AWS 上の置き場**と**デプロイの繰り返し**まで完了する。

**設計上含める必須要素:**

| 領域 | 内容 |
|------|------|
| **アカウント／環境** | 利用リージョン、環境名（`dev` / `prod` 等）、命名規則。 |
| **ネットワーク** | VPC、サブネット、Lambda の outbound、DB（Aurora）への到達性。 |
| **データストア** | Aurora Serverless v2（または採用する DB）と、マイグレーション適用の流れ（`db/migrations`）。 |
| **コンピュート** | 各ワーカー用 Lambda、EventBridge ルール、API 用 API Gateway ＋ Lambda（FastAPI / Mangum）。 |
| **設定・秘密** | Secrets Manager 等と環境変数の配線（リポジトリに秘密を置かない）。 |
| **デプロイ** | CDK `deploy`、および／または GitHub Actions 等による自動デプロイ、初回ブートストラップ手順。 |
| **観測性・運用** | CloudWatch ログ、失敗時の追跡、バックアップ・コストの方針メモ。 |

`infra/README.md` および `infra/stack.py` のスケルトンを、この波で実リソースに置き換える。

**注:** 第 2 波と第 3 波の前後は、チーム都合で第 3 波を前倒しする選択は可能だが、**デフォルトの合意順序は 1 → 2 → 3** とする。

---

## 4. 現状サマリ（ワーカー）

`workers/README.md` と実装の対応関係（要点のみ）。

| コンポーネント | 状態（概要） |
|----------------|--------------|
| `market-stats-refresh` | OAuth ＋ Browse API 連携、`market_stats` 更新。 |
| `sourcing-scan` | DB アクティブ行 ＋ `http_json` / `amazon` / `walmart` / `target`（テナントプロキシ URL または env テンプレート＋`external_product_id`）、イベント `custom`、SSRF・`last_fetched_at`。 |
| `inventory-sync-ebay` | `inventory_current` → eBay 数量更新、フラグで Sell Inventory API 呼び出し。 |
| `repricing` | 前提・ソーシング・市場 stats から固定価格更新。 |
| `orders-sync` | イベント `items[]` による upsert。`source: "ebay_fulfillment"` で Sell Fulfillment `getOrders` 同期（第 1 波・項番 1 完了）。 |

---

## 5. 横断関心

- **`shared/`:** ドメイン型・利益計算・バリデーションの共有化（重複排除）。  
- **`tests/`:** API・ワーカー共通のユニット／統合方針（`tests/README.md` と整合）。  
- **CI:** 既存 GitHub Actions（API / workers / web）を維持し、新規ワーカー・パッケージは同様に品質ゲートを通す。
